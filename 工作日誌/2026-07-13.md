### 🔄 C 重啟（使用者指示「C繼續」，推翻結案）
- **實作完成**（分支 experiment/worker-gpu，seg-worker.js）：
  - session 用 `preferredOutputLocation:'gpu-buffer'`，兩個輸出留在 GPU。
  - CPU 只用 `getData()` 下載小的 output0（~200KB）做 box 解碼+NMS。
  - 遮罩合成 sigmoid(coeffs·protos)≥0.4 改用自寫 WGSL compute shader 在 GPU 算（比 logit 門檻省 exp），每偵測只回讀 mh×mw 小遮罩（~16KB）→ 省掉 512KB protos readback。
  - 開關：`?noc` 關閉 C 做 A/B；shader 或 gpu-buffer 任一步失敗自動退回 CPU 路徑。
- **實測（桌機 Chrome、r4@256、`_bench/cbench.html`、30張取後25平均）**：
  - C開：total **24.8ms**（run 8.5 + o0下載等待 12.2 + shader+回讀 2.8），mode=gpuC
  - C關：total **26.5ms**（infer 25.1 含全部 readback）
  - **省 1.7ms（~6%）**；`dl` 12.2ms 主要是等 GPU 推論跑完，不是傳輸量 → 再把解碼/NMS搬GPU也省不了多少（上限已到）。
  - **正確性：GPU vs CPU 遮罩逐像素完全一致（mismatch 0）、分數相同** → 命中判定不受影響。
  - 手機上 readback 較貴，實際省多少要手機實測才知道。
- **附帶抓到兩個潛在地雷（順手修了）**：
  1. worker 內 ORT 動態 import `ort-wasm-simd-threaded.jsep.mjs` 會解析到本站 `/js/` 下 404（新版 Chrome 行為），造成 worker **無聲卡死 30 秒後靜默退回主線程**。修法：worker 里明確設 `ort.env.wasm.wasmPaths` 指向 CDN。⚠️ **此修應該也要進 main（production 在新版 Chrome/Android 可能同樣中招）**。
