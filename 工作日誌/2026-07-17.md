# 2026-07-17 工作日誌

## 議題：正確率不夠高，AI 能吃到多高畫質？

### 現況
- 偵測輸入固定 192×192（`?hq` 才 256），與相機畫質（4K）無關——偵測前先縮 384 縮圖再 letterbox。
- 現役 `seg_r9` 只匯出 128/192/256 三種 ONNX；要吃更高解析度需回 DGX 重新 `yolo export imgsz=…`（32 的倍數皆可）。

### 分析結論
- 模型訓練在 imgsz=256，推論解析度拉離訓練解析度正確率反而會掉。
- 自家評測（models/解析度正確率比較.md）：256 vs 192 的 mAP50 僅差 0.918 vs 0.881；r9 在 192/256 幾乎相同 → **瓶頸不在解析度，在模型與訓練資料**。
- 硬上 384 運算量是 256 的 2.25 倍（iPhone 連 256 都嫌重），預期收益極低且加重 GPU 爭用。

### 提高正確率的建議順序
1. 補訓練資料（漏抓/誤判場景，人工標註優先——使用者鐵則）。
2. 用 imgsz=320/384 重新「訓練」r10，再配套高解析度推論（代價：可能只有 ?hq 模式跑得動）。
3. 換大一號模型 yolo11s-seg（運算量約 3.6 倍，舊機出局）。

### 待使用者回覆
正確率不足的具體症狀：漏抓人？誤判非人？剪影邊緣不準？——決定走哪條路。

## 已實作：fps 夠好自動升一階偵測解析度（main.js）

使用者需求：手機 fps 能維持 16 以上，就自動幫他升一階 AI 畫質。

- 升階梯：128→192→256，與既有降級梯（256→192→128）成對。
- 觸發條件：**偵測 ≥18fps 且 畫面 ≥40fps，連續 3 秒**（單秒達標不算，防瞬間好轉就升；歷程 16/30→20/40→18/40，因偵測被 DET_MIN_INTERVAL=50ms 節流封頂在 ~20、實測跑順約 18–19，門檻 20 幾乎不可觸發，故降 18）。
- 防振盪：`upBanned` 記住「曾經降級的解析度」，之後不再自動升回去（升↔降只會發生一次）。
- 只在 WebGPU worker 路徑升階；WASM 不升（128→192 運算量翻倍會直接卡死）。
- `?hq` 開局即 256 不受影響；降級邏輯原樣保留。
- 版本號 v41→v42（index.html 全部 + seg-worker 引用）。

## 解析度階梯擴到 320/384（回答「256 也順能不能更高」）

**DGX 實測 r9 在官方 COCO 人工標註 valgt（2693 張）的多解析度成績：**

| 推論解析度 | mask mAP50 | 備註 |
|---|---|---|
| 192 | 0.503 | 現役預設 |
| 256 | 0.576 | ?hq |
| 320 | 0.616 | 🆕 |
| 384 | 0.645 | 🆕，+12% vs 256 |

- 結論顛覆先前認知（「訓練 256 推論拉高沒用」）：**推論解析度越高正確率持續上升**，小目標/多人場景收益最大。
- 已匯出 `seg_r9_320.onnx`/`seg_r9_384.onnx` 進 repo（gitignore 需 -f，循既有慣例）。
- `SEG_LADDER = [128,192,256,320,384]`（detector-seg.js），main.js 升降級改走階梯：
  - 升：偵測≥18 且畫面≥40 連續 3 秒升一階（僅 WebGPU）。
  - 降：畫面<24 或偵測<12 降一階；**>256 的重階多一條「偵測<15 就降」**——高解析正確率再高，偵測更新率掉太多剪影跟不上移動的人。
  - 曾降級的階不再升回（防振盪）。
- commit 7b16ad6，版號 v45。
