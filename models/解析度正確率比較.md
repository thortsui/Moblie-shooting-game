## 計算量怎麼變（回答「少 32 個步驟還是什麼」）
**不是固定少幾步，而是跟「解析度的平方（像素數）」成正比**：
- 256→192：像素 65536→36864，運算量 1.56→0.88 GFLOPs = **少 44%**（省 0.68 G）。因為 (192/256)² = 0.56。
- 192→128：再少 56%。256→128：只剩 1/4（少 75%）。
- 偵測候選點也同步：1344→756→336；遮罩格 64²→48²→32²。

## ⭐ 最關鍵發現：速度差多，要看跑在哪
- **WebGPU（iPhone/新機）**：解析度降低 **fps 幾乎不變**（40→47→49→50）！因為在這種小模型上，WebGPU 不是被「運算量」卡住，而是被**額外開銷**（資料傳輸 + JS 後處理）卡住 → **降解析度換不到多少速度**。
- **WASM（舊機無 WebGPU）**：fps 跟運算量成正比大幅變化（3→6→17→28）→ 降解析度**非常有感**。

## 結論
1. 你的 iPhone（WebGPU）：256 和 192 速度其實**差不多**（40 vs 47fps），真正的 fps 瓶頸是**後處理/資料傳輸開銷**，不是模型解析度 → 要更快得靠 **Web Worker/後處理優化**，不是繼續降解析度。
2. 舊機（WASM）：降解析度才有大幅加速。
3. 品質面 192 是實用下限（召回率 0.803），128 以下大量漏人。
4. **建議**：iPhone 其實可以用回 256（速度差不多、品質更好）；真要加速走 Web Worker。
