# 如何加快「中間傳輸/開銷」時間

建立 2026-07-12。背景：實測發現 WebGPU 上 fps 瓶頸不是模型運算，是**開銷**（資料在 CPU↔GPU 之間傳來傳去 + JS 後處理）。以下方法針對這個開銷。

## 開銷從哪來（一次偵測的資料流）
1. 相機影格 → 畫到小 canvas（縮 256）
2. **`getImageData` 把像素從 GPU 讀回 CPU** ← 慢（GPU→CPU 回讀）
3. 轉成 Float32Array → **上傳回 GPU** 給模型 ← 又一次傳輸
4. 模型輸出（含 32×64×64 遮罩原型）**從 GPU 讀回 CPU** ← 大量回讀
5. JS 迴圈解碼遮罩（乘加）← CPU 運算

## 方法（依 CP 值排序）

### 方法 A：降低偵測頻率（最easy，效果大）
每 2~3 影格才偵測一次，畫面/繪製維持 60fps，藍框更新慢一點（你已接受「藍框可延遲」）。
→ 開銷每秒直接砍半。**幾行程式、風險低。推薦先做。**

### 方法 B：免 getImageData 回讀（直接餵畫面給 GPU）
用 `ort.Tensor.fromImage(canvas/video/ImageBitmap)` 或 GPU 貼圖，讓 onnxruntime 直接吃畫面、在 GPU 內做正規化，省掉步驟 2、3 的 CPU↔GPU 來回。
→ 砍掉輸入端的傳輸。中等工程。

### 方法 C：輸出留在 GPU（io-binding）
設 `preferredOutputLocation:'gpu-buffer'`，模型輸出不回讀 CPU（省步驟 4 的大量回讀），但遮罩解碼要改用 WebGPU shader 在 GPU 上做。
→ 砍掉輸出端傳輸 + 後處理搬 GPU，**最徹底但最難**。

### 方法 D：偵測搬 Web Worker（保畫面流暢）
把推論+後處理搬到背景執行緒（OffscreenCanvas），主線程只顯示畫面。
→ 不減少單次傳輸，但**畫面永遠不卡**（符合你「畫面不能延遲」）。中等工程。

### 方法 E：優化 JS 遮罩解碼
只解碼命中/顯示需要的區域、降低遮罩解析度、typed-array 優化。
→ 減步驟 5 的 CPU 運算。小工程、小效果。

## 建議順序
1. **A（降頻）** — 先做，立即有感。
2. **B（免回讀輸入）** — 次之，砍輸入傳輸。
3. **D（Web Worker）** — 要「畫面絕對流暢」時做。
4. C/E — 進階微調。
